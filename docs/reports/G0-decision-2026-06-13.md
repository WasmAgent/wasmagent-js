## What gates G0

The plan defines a single, falsifiable threshold:

> Any ≤2B / ≤1.2GB model, under arm (e) full-stack, achieves
> `multi-turn-tool-exec` ≥ 50%, **AND** McNemar p<0.05 vs bare arm (a).

That number is not negotiable — it's the line below which "the
scaffold is doing real work" cannot be defended. The plan also makes
the failure branch explicit and concrete (no ambiguity about what
"fail" means): the report goes back to the evomerge / SFT track,
retest in 3 months. **In either branch, V1 + V3 are not sunk cost.**

### Three things this n=90 read settles

1. **The 50% Run D number was noise.** Standard "small-n point
   estimate is unreliable" — the lesson is "do not promote G0 status
   on a Wilson CI that includes the threshold". Run E's CI [3.8, 15.2]
   on the best (1.5B, arm e) pair sits entirely below 50%. The
   verdict box flip is unambiguous.

2. **Arm (e) over arm (c) buys 1.1pp at 5× wall-clock.** SC k=5
   majority on terminal state is honest signal but the cost-per-pp
   is so far above the consumer-laptop budget that the arm doesn't
   change the qualitative read. The CodeAct arm (c) is the cheap
   ceiling; SC adds noise reduction that can't lift a model off
   the floor.

3. **The 0.5B → 1.5B gap is ~6× on accuracy at ~5× wall.** Cost-
   wise the larger model is the strict winner; size economics are
   not the bottleneck. The bottleneck is *which model family* —
   evomerge's GSM8K-tuned 1.5B is in the same accuracy band as
   off-the-shelf instruction-tuned models on this multi-turn surface
   despite its math gain. Confirms the TinyLLM 2025-11 thesis:
   single-turn ≠ multi-turn.

### Three things this read surfaced (that the production grid will
either confirm or reject)

1. **GSM8K 70.5% does not transfer to multi-turn agent loops.**
   `evomerge-qwen25-1b5` is the same Q8_0 weights that beat
   `qwen2.5:0.5b` four-to-one on GSM8K, yet it scored *worse* on
   every arm except (e). Hypothesis: SFT on math-only data
   degrades the model's ability to emit structured tool_use blocks
   in a sustained loop. This directly mirrors the TinyLLM 2025-11
   paper's central claim (single-turn benchmarks decouple from
   multi-turn agent ability) — so the right next step on the
   evomerge side is a multi-turn SFT pass, not more GSM8K data.
   In the failure-branch path the plan already names xLAM-2 1B
   (~8% → 35% multi-turn after targeted SFT) as the precedent.

2. **CodeAgent (arm c) misfires on small instruct models.**
   The stderr was full of `[CodeAgent] extractCode: no code block
   found in response: <model says it cannot run code in this
   sandbox>`. The default `CodeAgent` system prompt assumes the
   model knows what a sandboxed JS execution context means;
   small instruct models interpret "sandbox" as "no DOM →
   refuse". This is a *prompt engineering* gap, not a kernel
   gap, and is fixable inside agentkit-js without retraining.
   Filed mentally as: "investigate `CodeAgent.DEFAULT_SYSTEM_PROMPT`
   on ≤2B models; consider few-shot the prompt with a worked
   tool_call → callTool() example."

3. **Arm (e) wall-clock for 1.5B = 52 seconds p95 per item.**
   k=5 SC × CodeAgent × grammar amortises to ~10 s per rollout.
   For an interactive desktop product (D1), that's the upper
   bound of "tolerable" — the user notices. If G0 ever passes
   on 1.5B, the killer-task latency budgets (D2 DoD) will need
   independent validation; arm (e) numbers here cannot be lifted
   verbatim.

## Run F final read (2026-06-13, n=90 per cell, 3 seeds × 30 items)

> Run F introduced arm (f) param-only — a tighter scaffold designed
> from the failure analysis of arm (c). Per the user direction "let
> the model only fill arguments, don't even let it choose tools",
> arm (f) uses two passes per step: pass 1 grammar-pins a tool-name
> choice from a closed enum, pass 2 grammar-pins the args for that
> exact tool. 1pass comparator combines both into a single oneOf-
> shaped grammar call.

### Headline numbers

| Model | bare (a) | param-only (f, 2-pass) | param-only-1pass | McNemar (f vs bare) |
|---|---:|---:|---:|---|
| `qwen2.5:0.5b`           | 11/90 (12.2%, [7.0, 20.6]) | 13/90 (14.4%, [8.6, 23.2]) | 11/90 (12.2%) | p=0.69 |
| `evomerge-qwen25-1b5`    | 0/90 (0.0%, [0, 4.1])     | **22/90 (24.4%, [16.7, 34.2])** | 7/90 (7.8%) | **p=4.8e-7** ⭐ |

### What the lift means

McNemar on (1.5B, arm-f) vs (1.5B, bare): **arm-wins=22, bare-wins=0,
both=0, neither=68**. Every paired cell where bare failed and arm-f
passed — the scaffold contributes a real, statistically robust 22
items. p<1e-6 by exact binomial. This is the cleanest "scaffolding
does non-trivial work" signal V2 has produced.

But: Wilson upper bound 34.2% is 16pp short of the 50% G0 floor.
Verdict box flips from FAIL-OOM (Run E) back to FAIL-improvable.

### Three findings

1. **Path 2 (this approach) is partially validated, not closed.**
   The +24.4pp absolute lift on 1.5B is meaningful but does not
   close the 50% gap on its own. Lift mechanism: arm-a generates
   structurally-malformed tool calls that the bare loop's
   parser/retry path can't recover from; arm-f's grammar makes
   structurally-malformed calls impossible by construction, so
   the model's per-call success = the model's per-call semantic
   correctness, not its JSON-syntax correctness.

2. **0.5B does not benefit (12.2% → 14.4%, p=0.69).** The
   intervention requires a model that can actually fill arguments
   correctly; 0.5B's failures shift from "wrong format" to "wrong
   semantics" and the grammar can't fix the latter.

3. **2-pass beats 1-pass on 1.5B (24.4% vs 7.8%).** Contradicts
   the prior "1-pass should win on cost". The mechanism is in the
   pass-2 prompt: when the model is asked "fill args for move_file"
   with the chosen tool name as an attention anchor, it produces
   correct args at a much higher rate than when the schema requires
   it to make both decisions in one shot. This is a useful design
   note for any future scaffolded-agent work — split decisions
   helps small models even when the schema would technically allow
   the joint form.

### Combined-path estimate (for the G0 PASS branch)

- Run E established the bare-stack ceiling at ~7.8% on 1.5B
- Run F establishes scaffold-only ceiling at ~24.4% on 1.5B (+16.6pp)
- xLAM-2 1B (literature) shows targeted multi-turn SFT lifts a 1B
  model from ~8% to ~35% (+27pp) on similar BFCL-v3-style multi-turn
  benchmarks
- If those lifts compose roughly additively, scaffold + multi-turn
  SFT on a 1.5B model could reach ~50% — the G0 PASS threshold
- 3B+ off-the-shelf models (e.g. xLAM-2-3b) reach 55.62% multi-turn
  per the TinyLLM 2025-11 paper, well above the threshold

These are projections, not measurements. Either branch (SFT or 3B+)
is now a reasonable next step; arm (f) gives the scaffold-side
contribution a quantified floor.

### What this tells us about scaling

- **0% → 32.2% lift on 1.7B (+32.2pp)**, vs **0% → 24.4% on 1.5B (+24.4pp)**.
  Both bare baselines are 0/90, so the scaffold contribution is the
  whole accuracy. Larger models inside the ≤2B band do better —
  consistent with TinyLLM 2025-11's gradient.
- **Wilson upper bounds: 1.5B → 34.2%, 1.7B → 42.4%.** Linear
  extrapolation suggests a ~2.4B model might cross 50% on the upper
  bound; but extrapolation from two points across a quantization
  boundary (1.5B Q8 vs 1.7B Q3) is unreliable. The next data point
  has to be measured, not projected.
- **Wall-clock cost**: 1.5B p95=18.6s/cell, 1.7B p95=72s/cell (~4×).
  8B (Run G-smoke single-cell sample, n=6) showed p95~135s. Going
  to 8B is a ~7× wall multiplier vs 1.5B. Not just an accuracy
  question — interactive desktop-agent latency budget breaks past
  ~30s/step.

### CI overlap with Run F's 1.5B result

The 1.5B [16.7, 34.2] and 1.7B [23.5, 42.4] confidence intervals
overlap substantially. We **cannot** claim 1.7B is statistically
significantly better than 1.5B on this n=90 read — we'd need a
direct paired test (same items, same seeds, both models) to make
that claim. What we CAN claim: both clear the bare baseline by
McNemar p<1e-6, both fall short of the 50% G0 upper-bound floor
by 8-16pp.

### Three things this n=90 settles

1. **Path 3 within ≤2B does not unlock G0.** Both 1.5B (Q8_0,
   1.6GB) and 1.7B (Q3_K_M, 0.9GB) leave the upper Wilson bound
   below 50%. The gap is narrower at 1.7B (8pp) than at 1.5B
   (16pp), but neither crosses the threshold.

2. **Going past ≤2B (8B+) is the next data point.** Run G-smoke
   on `evomerge-qwen3-v2` (8B Q3_K_M) showed 4/6 (66.7%, [30.0,
   90.3]) at n=6 — directional signal that 8B might cross 50%
   on n=90, but Run D taught us not to trust n=6 numbers near
   the threshold. A full n=90 run on 8B costs ~3-4h wall on this
   host; deferred until either (a) we explicitly want that data
   point or (b) parallel SFT work lands.

3. **Scaffold contribution does scale with model.** The +32.2pp
   on 1.7B from the SAME prompt + grammar that gave +24.4pp on
   1.5B suggests the scaffold isn't saturating at this size —
   bigger models can use more of what the scaffold provides.
   Useful evidence for combining path 2 (scaffold) with path 1
   (SFT) — they're not redundant levers.

  that learning" — there's no native tool_use to override; bare on
  this architecture (Qwen3-1.7B, q3_k_m / q4_k_m) just doesn't
  produce structurally legal tool_calls without grammar at all.

**Run G vs Run H baseline drift:**
- Same model (`evo-qwen3-1b7-q3km`) on arm-f: Run G = 32.2%, Run H = 40.0%
- Same items, same seeds — the 8pp gap is from temperature=0
  determinism breaking down inside Ollama under different VRAM
  preconditions (the Run G machine had recently unloaded a different
  model; Run H was a cold load). This is a real reproducibility
  caveat for any future scaffold work — n=90 is enough for
  intra-run statistical power but not enough to anchor headline
  numbers across runs without explicitly fixing the warm-up state.

### Diagnostic conclusion

The training-data-vs-scaffold mismatch is the cause. SFT data trains
the assistant to emit `tool_calls` in a single message
(`assistant content + tool_calls array → tool result → assistant
DONE`). arm-f's two-pass grammar forces the model to first emit
`{"choice":"<tool_name>"}` then `{<args>}` — a sequence the SFT'd
model has never seen. The model overfits the SFT pattern (loss
0.005 by step 100, severe overfit on 1600 similar items) and then
gets blocked by arm-f's grammar from emitting that pattern.

### What the G1 v3 20% data point actually tells us

**It is NOT G0 progress.** v3's 18.9% is *below* the unmodified base's
40.0% — the SFT made things worse for the production deployment
shape (arm-f). To use SFT productively under arm-f, the training
data must be in arm-f's two-pass shape, not the native tool_calls
shape.

### Required fix for the next SFT iteration

The dataset generator at
`agentkit-js/datasets/multi-turn-sft-starter/generate.mjs` produces
the **wrong shape** for arm-f training. Two options:

1. **Regenerate the SFT data in arm-f's two-pass format** — every
   record's `messages` should have an `assistant` turn with
   `content: '{"choice":"move_file"}'` followed by the next
   `assistant` turn with `content: '{"from":"...", "to":"..."}'`,
   with the `tool` role in between. Two model-step events per
   tool call instead of one. This trains the model to emit exactly
   what arm-f grammar requires.

2. **Use arm-a (bare) at production time, not arm-f** — and add a
   different scaffold layer that's compatible with native
   `tool_calls`. arm-a's 0/90 on this model says native tool_use is
   too unstructured to be the production path on Qwen3-1.7B
   regardless of SFT.

**Recommendation: Option 1.** arm-f at 40.0% on baseline gives a real
scaffold lift; the SFT data shape is the fixable component.
Estimated work: ~2 hours regenerating the dataset + 16 hours
re-running SFT.

### Updated G0 status

Still **FAIL — close but improvable**. The Run G ceiling (1.7B arm-f
32.2%, Run H baseline 40.0%) and the SFT lift potential (xLAM-2 1B
+27pp, IF data is shape-aligned) jointly suggest 1.7B arm-f could
plausibly reach the 50% G0 floor with one more SFT iteration in the
right shape. But the current v3 explicitly does not get us there.
