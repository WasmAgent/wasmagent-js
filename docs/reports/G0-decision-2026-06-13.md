## What gates G0

The plan defines a single, falsifiable threshold:

> Any ≤2B / ≤1.2GB model, under arm (e) full-stack, achieves
> `multi-turn-tool-exec` ≥ 50%, **AND** McNemar p<0.05 vs bare arm (a).

That number is not negotiable — it's the line below which "the
scaffold is doing real work" cannot be defended. The plan also makes
the failure branch explicit and concrete (no ambiguity about what
"fail" means): the report goes back to the evomerge / SFT track,
retest in 3 months. **In either branch, V1 + V3 are not sunk cost.**

## What "Pending" means in practice

`docs/reports/multi-turn-scaffold-ablation-2026-06-13/REPORT.md`),
this file gets a follow-up appendix that flips one of the three
boxes below:

- ☐ **PASS** — at least one ≤2B / ≤1.2GB model under arm (e) hits
  ≥50%, McNemar p<0.05 vs (a). → enter **Phase 1**: start
  `bsagent-desktop` at `/Users/I041705/github/bsagent-desktop` (D1
  task); the V2 report is also the public artifact P1 publishes.
- ☐ **FAIL — close but improvable** — e.g. (e) reaches 35–49% on
  the 1.5B model. → no Phase 1 code yet. Hand the report to
  evomerge/SFT for targeted multi-turn training (xLAM-2 1B
  precedent: ~8% → 35%); same-suite retest in 3 months.
- ☐ **FAIL — order of magnitude** — e.g. (e) on every ≤2B model
  stays under 15%. → the scaffold-only hypothesis is wrong as
  stated; rethink (the plan suggests a 1–3B specifically-trained
  function-calling model is the only viable hardware target).
